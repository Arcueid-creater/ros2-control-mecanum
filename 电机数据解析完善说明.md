# 电机数据解析完善说明

## 问题描述

原来的`parseMotorData`函数只是将解析出的数据存储到`motor_data_buffer_`中，但没有将数据复制到电机状态变量中，导致控制器无法读取到实际的电机反馈数据。

## 修改内容

### 添加完整的数据解析逻辑

在CRC校验通过后，添加了以下功能：

1. **按协议格式解析数据**
   - 每个电机数据包：17字节
   - 格式：`motor_id(1字节) + position(4字节) + velocity(4字节) + current(4字节) + temperature(4字节)`

2. **更新电机对象状态**
   ```cpp
   motors_[i]->angle_current = position;
   motors_[i]->measure.speed_aps = velocity;
   motors_[i]->measure.real_current = static_cast<int16_t>(current);
   motors_[i]->measure.temperature = static_cast<uint8_t>(temperature);
   ```

3. **更新状态接口变量**（供ros2_control控制器读取）
   ```cpp
   state_positions_[i] = position;
   state_velocities_[i] = velocity;
   state_currents_[i] = current;
   state_temperatures_[i] = temperature;
   ```

4. **更新通信状态**
   ```cpp
   motors_[i]->update_timestamp(rclcpp::Clock().now());
   motors_[i]->status = MOTOR_OK;
   ```

## 数据流程

```
串口接收 → motor_rx_buffer_ 
         ↓
    查找帧头 0xFF 0xAA
         ↓
    读取数据长度
         ↓
    验证CRC32
         ↓
    提取纯净数据 → motor_data_buffer_
         ↓
    按协议解析（每17字节一个电机）
         ↓
    更新电机状态变量
         ↓
    控制器通过state_positions_等读取
```

## 协议格式

### 下位机发送格式

```
+--------+--------+--------+--------+------------------+--------+
| 0xFF   | 0xAA   | Length | Length | Motor Data       | CRC32  |
|        |        | (Low)  | (High) | (N × 17 bytes)   | (4B)   |
+--------+--------+--------+--------+------------------+--------+
```

### 单个电机数据格式（17字节）

```
+----------+----------+----------+----------+-------------+
| Motor ID | Position | Velocity | Current  | Temperature |
| (1 byte) | (4 bytes)| (4 bytes)| (4 bytes)| (4 bytes)   |
+----------+----------+----------+----------+-------------+
```

## 示例

### 3个电机的数据包

假设有3个电机（ID: 1, 2, 3），数据包结构：

```
帧头: 0xFF 0xAA
长度: 0x33 0x00 (51字节 = 3 × 17)
数据:
  电机1: ID=1, pos=1.57, vel=0.5, cur=2.3, temp=45.5
  电机2: ID=2, pos=0.78, vel=0.3, cur=1.8, temp=42.0
  电机3: ID=3, pos=2.35, vel=0.8, cur=3.1, temp=48.2
CRC32: 0xXXXXXXXX
```

总长度：4(帧头+长度) + 51(数据) + 4(CRC32) = 59字节

## 注意事项

1. **电机ID匹配**：使用`config_.tx_id`来匹配电机，确保下位机发送的motor_id与配置文件中的tx_id一致

2. **数据类型转换**：
   - `current`：float → int16_t
   - `temperature`：float → uint8_t

3. **时间戳更新**：每次接收到数据都会更新时间戳，用于检测通信超时

4. **状态标记**：成功解析后将电机状态设置为`MOTOR_OK`

## 下位机实现参考

下位机需要按照以下格式发送数据：

```c
// 构建数据包
uint8_t buffer[100];
size_t offset = 0;

// 帧头
buffer[offset++] = 0xFF;
buffer[offset++] = 0xAA;

// 数据长度（3个电机 × 17字节 = 51字节）
uint16_t data_length = 51;
memcpy(buffer + offset, &data_length, 2);
offset += 2;

// 电机1数据
buffer[offset++] = 1; // motor_id
float pos1 = 1.57f, vel1 = 0.5f, cur1 = 2.3f, temp1 = 45.5f;
memcpy(buffer + offset, &pos1, 4); offset += 4;
memcpy(buffer + offset, &vel1, 4); offset += 4;
memcpy(buffer + offset, &cur1, 4); offset += 4;
memcpy(buffer + offset, &temp1, 4); offset += 4;

// 电机2、3数据...（同样格式）

// 计算并添加CRC32
uint32_t crc = calculateCRC32(buffer, offset);
memcpy(buffer + offset, &crc, 4);
offset += 4;

// 发送
serial_write(buffer, offset);
```

## 测试建议

1. **单电机测试**：先测试单个电机的数据接收和解析
2. **多电机测试**：逐步增加电机数量，验证解析逻辑
3. **数据验证**：在Foxglove或rviz中查看电机状态是否正确更新
4. **超时测试**：断开串口连接，验证电机状态是否变为`MOTOR_LOST`
